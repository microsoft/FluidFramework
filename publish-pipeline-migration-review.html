Why the direct ESRP pipeline should work

Artifact and ordering checks

Build Client remains the only packer. The publish pipeline consumes its exact pack artifact, validates the metadata and public order file, rejects missing, duplicate, or private packages, and groups tarballs by dependencies.

writes zero-padded numeric directories and a manifest. Every dependency selected for publication is assigned to an earlier layer before the ESRP helper sees it.

Offline release runtime

A normal job authenticates to the internal npm feed and materializes the helper plus production dependencies as a pipeline artifact. The production release job downloads that artifact instead of trying to install packages after network isolation is enabled.

and the checked-in JWS bigint patch make the temporary fork runnable with Node 24.

Credential and release checks

Staging storage uses the refreshable WIF session maintained by AzureCLI@2. ESRP authentication and request signing use two separate PFX secrets from AzureKeyVault@2. The helper receives every required environment value defined by .

Failure and retry checks

The upstream FluidFramework commit keys release state, each layer is marked only after ESRP reports success, and the release step retries three times. Git tagging still depends on the release deployment, so a partial or failed publication cannot create a successful release tag.

Remaining live gates: replace every *_REPLACE_ME value, authorize the two Azure service connections and GitHub connection, assign staging Blob Data Contributor + Delegator roles, and run an Azure DevOps YAML preview followed by a non-production end-to-end release. Static checks cannot validate those external resources.
Detailed file-by-file change log: what changed and why

Compared with origin/main, the branch changes 20 files: 12 new publishing pipeline files and eight direct-ESRP-helper changes.

End-to-end branch flow

  1. Internal Build - client packages completes and triggers the ff_publishing pipeline.
  2. The exact pack artifact and publish-metadata.json are downloaded and validated.
  3. Validated metadata selects the internal feeds, public npm publication, tagging, and manifest behavior.
  4. Internal packages continue through flub publish tarballs.
  5. Public tarballs are converted into dependency-topological numeric layers.
  6. The direct ESRP helper publishes each layer sequentially through the ESRP Release API.
  7. Successful public releases are tagged, and eligible publications generate and upload manifests.

Pipeline entry and orchestration

What it does: Declares the 1ES template, FluidFramework GitHub resource, Build Client and Build Tools pipeline resources, internal feed settings, ESRP and staging configuration, and manifest storage. It enables the Azure AD, Key Vault, ARM, Storage, and GitHub network destinations required by production release jobs.
Why: Build Client is now artifact-only; this repository needs an explicit composition root for every external publishing dependency.


Purpose: Trust boundary between Build Client and publishing.
What it does: Downloads the triggering run's artifact; validates schema version, release type, semantic version, typed booleans, release group, tarballs, and all package-order files; and exports named stage outputs. Source branch and commit come from the client pipeline resource.
Why: This pipeline's Build.* values describe ff_publishing, so they cannot safely drive FluidFramework release decisions.


Purpose: Stage-level publishing orchestrator.
What it does: Creates internal-test, internal-build, internal-dev, public npm, and manifest stages. Runtime conditions consume the validated metadata outputs rather than pipeline-global variables.
Why: Building occurs upstream, so this pipeline needs only explicit publication eligibility and dependency rules.

Internal Azure Artifacts path


Purpose: Protected internal-feed deployment jobs.
What it does: Downloads client and Build Tools artifacts, installs flub from packed Build Tools tarballs, and invokes the internal publication steps. Each feed has mutually exclusive production and non-production jobs.
Why: 1ES requires templateContext.isProduction to be a compile-time literal, while release classification is available only at runtime.


Purpose: Actual internal npm-feed publication and promotion.
What it does: Creates and authenticates a feed-specific .npmrc, selects rc/latest/dev, runs flub publish tarballs with retries, promotes full internal-build releases, and always removes credentials.
Why: Internal feeds remain on the established flub path, while credential logging and cleanup behavior needed hardening.

Public ESRP path


Purpose: Complete public npm release path.
What it does: Builds an offline Node 24 helper artifact, consumes that artifact with the upstream pack artifact, creates dependency layers, preserves npm tags, downloads two PFX secrets, and runs the helper inside AzureCLI@2 with a renewable WIF session, retries, and a six-hour timeout. Tagging depends on successful completion.
Why: The direct API helper replaces EsrpRelease@12, requires layered packages and two certificates, and cannot install dependencies from a network-isolated production job.


Purpose: Convert Fluid's flat tarballs into the helper's input format.
What it does: Extracts package metadata, rejects private, missing, or duplicate packages, builds a graph from dependencies/optional dependencies/peer dependencies, detects cycles, writes zero-padded topological layers, and emits layer-manifest.json.
Why: Public packages must not become visible before package versions they reference are available.


Purpose: Create an installable production runtime for the helper.
What it does: Copies production source, excludes tests and fixtures, copies runtime dependencies, replaces the Yarn patch: dependency with plain jws@4.0.1, and writes a standalone package manifest.
Why: The temporary fork was copied from the larger Beachball Yarn workspace and is not independently installable as committed.


Purpose: Preserve the upstream JWS compatibility fix.
What it does: Adds bigint serialization while retaining numeric JSON output.
Why: ESRP's JWS expiration is expressed as .NET ticks in a JavaScript bigint; unpatched JWS fails before request submission.

Git tagging and release manifests


Purpose: Tag the released FluidFramework commit.
What it does: Creates and pushes client_v<version> using credentials persisted by the FluidFramework GitHub checkout.
Why: A release tag must identify the exact upstream commit and must not be created unless public ESRP publication fully succeeds.


Purpose: Generate Fluid client release manifests.
What it does: Checks out the exact upstream commit, installs and builds Fluid Build Tools, generates full and version-specific reports, and forwards them to the upload template.
Why: Manifests must describe the source that produced the packages, not the ff_publishing checkout.


Purpose: Persist release reports.
What it does: Copies reports into artifact staging, uploads them to Azure Blob Storage with a short cache TTL, cleans working directories, and publishes the report artifact.
Why: Storage account, container, connection, and artifact names need to be explicit cross-repository dependencies.

Direct ESRP helper changes


Purpose: Operational contract for the helper.
What changed: Documents staging RBAC, layered packages, offline runtime artifacts, Key Vault requirements, and the long-running keepAzSessionActive/STAGING_USE_AZURE_CLI pattern while retaining the legacy static mode.
Why: Consumers need the renewable-token pattern instead of copying a capture-once WIF assertion that expires during long releases.


Purpose: Typed release configuration.
What changed: Models staging authentication as a discriminated union: Azure CLI mode requires only the storage account, while legacy static federation requires client ID, tenant ID, and ID token.
Why: Invalid authentication combinations are rejected at compile time and runRelease can narrow without non-null assertions.


Purpose: Parse and validate the process environment.
What changed: Recognizes STAGING_USE_AZURE_CLI, conditionally requires the correct staging fields, retains ESRP contact fallback, and reports all missing variables together.
Why: The pipeline/helper boundary must match the selected auth mode exactly.


Purpose: Renewable Azure Storage token provider.
What it does: Calls az account get-access-token, normalizes both Azure CLI expiration formats, and returns the Storage SDK AccessToken shape.
Why: A single WIF assertion cannot safely support a release lasting multiple access-token lifetimes.


Purpose: Main layer-release orchestration.
What changed: Selects Azure CLI token acquisition for staging Blob Storage while retaining the static-WIF fallback. Existing behavior loads release state, skips completed layers, zips each layer, submits it, waits for success, and marks it published afterward.
Why: Blob upload, SAS creation, state writes, and cleanup can occur hours after the job starts.


Purpose: Complete helper test fixtures.
What changed: Explicitly marks the existing fixture as legacy static authentication.
Why: Tests need to exercise a specific discriminated-union branch.


Purpose: Verify the environment contract.
What changed: Adds coverage proving Azure CLI mode does not require static staging identity values, alongside contact fallback, tag, and missing-variable coverage.
Why: This prevents drift between pipeline variables and helper validation.


Purpose: Verify the Azure CLI token adapter.
What it tests: Token extraction, epoch-seconds-to-milliseconds conversion, and rejection of incomplete responses.
Why: Incorrect expiration handling could cause the Storage SDK to cache an expired token or refresh at the wrong time.

External producer dependency: This branch does not build or pack Fluid packages. The FluidFramework Build Client pipeline must produce publish-metadata.json, four feed-specific packagePublishOrder-*.txt files, and tarballs/*.tgz in the pack artifact.
