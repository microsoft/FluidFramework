## Property Details

### entryPoint {#entrypoint-propertysignature}

Exposes a handle to the root object / entryPoint of the data store. Use this as the primary way of interacting with it. If this property is undefined (meaning that exposing the entryPoint hasn't been implemented in a particular scenario) fall back to the current approach of requesting the root object through the request pattern.

#### Signature {#entrypoint-signature}

```typescript
readonly entryPoint?: IFluidHandle<FluidObject>;
```

#### Remarks {#entrypoint-remarks}

The plan is that eventually the data store will stop providing IFluidRouter functionality, this property will become non-optional and return an IFluidHandle (no undefined) and will become the only way to access the data store's entryPoint.

### IFluidRouter {#ifluidrouter-propertysignature}

{{% callout warning Deprecated %}}
- Will be removed in future major release. Migrate all usage of IFluidRouter to the "entryPoint" pattern. Refer to Removing-IFluidRouter.md

{{% /callout %}}

readonly IFluidRouter: IFluidRouter;

### request {#request_1-methodsignature}

{{% callout warning Deprecated %}}
- Requesting an arbitrary URL with headers will not be supported in a future major release. Instead, access the objects within the DataStore using entryPoint, and then navigate from there using app-specific logic (e.g. retrieving a handle from a DDS, or the entryPoint object could implement a request paradigm itself)

NOTE: IDataStore.request(\{url: "/"\}) is not yet deprecated. If and only if the DataStore implementation supports it, that overload may be used as a proxy for getting the entryPoint until [entryPoint](/docs/apis/runtime-definitions/idatastore-interface#entrypoint-propertysignature) is mandatory.

Refer to Removing-IFluidRouter.md for details on migrating from the request pattern to using entryPoint.
